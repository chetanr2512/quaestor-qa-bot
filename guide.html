High Level

System Architecture Overview

The Autonomous QA Agent is an end-to-end testing pipeline. It ingests human-readable requirements, uses Claude 3.5 Sonnet to autonomously generate test plans, executes them directly against a live web browser, and reports the findings to a dashboard and Jira.

Core Philosophy

Rather than writing brittle, hardcoded Cypress or Selenium scripts, this architecture treats the application exactly how a human QA engineer would. It relies on Visual DOM Understanding and Goal-Oriented Action Execution.

Code Level

Pipeline Initialization

The entire pipeline kicks off in agent/main.py. This file acts as the orchestrator for the modular components.

1. Argument Parsing & Ingestion

The CLI accepts arguments defining the source of truth (e.g., Jira or Sheets) and fetches the raw text descriptions of what needs to be tested.

if args.source == "jira":
    client = JiraClient()
    tickets = client.fetch_tickets(jql=args.source_id)
elif args.source == "sheets":
    client = SheetsClient()
    tickets = client.fetch_tickets(sheet_id=args.source_id)

2. AI Test Generation

The raw tickets are passed to the TestGenerator. This utilizes LangChain to send a prompt to Claude, asking it to break down the ticket into logical TestCase Pydantic models containing specific steps and assertions.

class TestCase(BaseModel):
    id: str
    name: str
    type: Literal["frontend", "backend"]
    steps: List[str]
    assertions: List[str]

3. Database Registration

Before execution begins, the pipeline registers a TestRun in Supabase to track total tests, duration, and cost. This allows the Next.js dashboard to instantly show a "Running" status.

The Magic

The Browser-Use Engine

The heart of the system is the BrowserRunner class, which implements the browser-use open-source library to bridge the gap between LLMs and physical browsers.

sequenceDiagram participant B as BrowserRunner participant BU as Browser-Use Agent participant P as Playwright (Chromium) participant C as Claude LLM B->>BU: execute(task_prompt) loop Autonomous Loop BU->>P: Inject JS to extract DOM P-->>BU: Returns Indexed Interactive Elements BU->>C: Send Prompt + DOM State C-->>BU: Returns JSON Action (e.g., click index 14) BU->>P: Execute native click on element P-->>BU: Await network idle end BU-->>B: Return AgentHistoryList (success/fail)

How it Works Under the Hood

  • DOM Tagging: The engine cannot simply "click the login button" because LLMs don't have spatial mouse coordinates. Instead, the engine scans the HTML using Playwright and injects a temporary visual overlay, assigning a unique ID to every interactive element (e.g. [14] <button>Login</button>).
  • Prompting the Model: Claude receives a text representation of the screen: "There is a button with ID 14 that says Login".
  • JSON Action Execution: Claude returns a structured tool call like {"action": "click", "index": 14}. The engine maps index 14 to its exact X/Y coordinate or XPath in Playwright and executes the physical click.

Execution Isolation

To prevent browser crashes across multiple tests, the BrowserRunner instantiates a completely fresh Chromium window per test case, closing it in the finally block to ensure a clean state and zero cache pollution.

Ecosystem

Integrations & Output

1. Supabase & The Next.js Dashboard

All test results are streamed into a PostgreSQL database managed by Supabase. The Next.js dashboard uses Supabase Realtime (WebSockets via supabase.channel('postgres_changes')) to instantly update the UI without needing to refresh the page.

// Dashboard Realtime Subscription
supabase.channel('test_runs_changes')
  .on('postgres_changes', { event: '*', schema: 'public', table: 'test_runs' }, fetchRuns)
  .subscribe();

2. Jira API Reporter

If the pipeline was triggered via a Jira JQL string, the Reporter class aggregates the pass/fail arrays. It maps each test case back to its parent Ticket ID and utilizes the Atlassian Document Format (ADF) API to automatically post a structured comment to the Jira Agile board.

3. Google OAuth 2.0

For Google Sheets, the pipeline utilizes the InstalledAppFlow.from_client_config methodology. Instead of unsecure service accounts, it dynamically spins up an HTTP server on localhost:8080 to catch OAuth callbacks, securely generating and caching a token_sheets.json file locally.
